# jarvis-os v0.3 · UI architecture · replanteo de scope

**Date:** 2026-04-30
**Branch:** `jarvis-arch-os`
**Status:** approved (brainstorming) → ready to plan
**Supersedes:** `2026-04-29-jarvis-os-extension-layer-design.md` (a archivar)
**Supersedes:** narrativa de `jarvis-os-spec-v0.2.docx.md` que enmarcaba el proyecto como "OS agéntico"

## Contexto y motivo del replanteo

La spec v0.2 (abril 2026) y el brainstorm del extension layer (2026-04-29) trataban a jarvis-os como un sistema operativo agéntico con capas externas (Linux MCP server, voice daemon, HUD overlay) ensambladas alrededor de IronClaw vía configuración + MCP, deliberadamente **sin forkearlo**. Esa premisa caducó cuando el repo `github.com/horelvis/jarvis-os.git` empezó a divergir de upstream con trabajo propio en `jarvis-arch-os`: ya es fork operacional con voice daemon, arch install scripts, y modificaciones in-tree.

Tres observaciones del 2026-04-30 fuerzan el replanteo:

1. **Las 8 tools de `jarvis_linux_mcp` (2475 LoC)** se cubren con `shell` builtin de IronClaw o se pueden reescribir como tools nativas del fork. El MCP server local agrega IPC overhead, una systemd unit (`jarvis-mcp-register`), boilerplate JSON-RPC stdio y `ironclaw mcp add ...` runtime registration sin aportar aislamiento real (los adapters son shellouts, no FFI).

2. **El chat ya existe** en IronClaw (REPL `jarvis-chat`, gateway web). Reimplementarlo en jarvis-os es trabajo redundante. Lo que aporta jarvis-os es **voz + UI visual futurista**, no una alternativa de chat.

3. **Computer-use que el user mencionó** corresponde a `browser` automation (FEATURE_PARITY: ✅ OpenClaw, ❌ IronClaw, P3). Es una feature concreta a portar como tool builtin del fork, no una capa nueva del SO. Scope acotado a navegación browser; control desktop-nativo queda para v2+.

El producto ahora se redefine como: **una UI desktop futurista (jarvis_ui) sobre IronClaw como motor agéntico headless, con extensiones in-tree del fork**. Los scripts Arch quedan como recipe opcional, no producto.

## Premisas redefinidas (no re-debatir)

- **jarvis-os = UI sobre IronClaw**, no sistema operativo.
- **Extensiones van al motor de IronClaw** (tools builtin/WASM/skills/channels), **no al filesystem del sistema**.
- **MCP local fuera**: `jarvis_linux_mcp` se desmonta como server; las tools sistémicas pasan a builtin del fork (lectura 1 del replanteo).
- **Browser automation**: portar de OpenClaw como tool builtin, P3 → priorizado para v1.x. Scope = browser, no desktop nativo.
- **Voice daemon**: Rust + ElevenLabs Conversational AI cloud, ya funciona, sin cambios v1.
- **Stack UI**: **Quickshell + QML puro**. Escape hatch a `QQuickItem` C++ solo si el ring no rinde (decisión empírica, no a priori).
- **Compositor**: Wayland wlroots-based (Hyprland, sway, river). Sin GNOME/Mutter (no soporta layer-shell). Sin X11.
- **arch/install.sh**: recipe opcional para Arch. Jarvis no requiere Arch ni Hyprland específicamente.

## Arquitectura general

Cuatro componentes, comunicación 100 % loopback localhost (excepto ElevenLabs cloud):

```
                       ┌──────────────────────┐
                       │      jarvis_ui       │  ← orbe + 3 widgets
                       │  Quickshell + QML    │
                       │   layer-shell        │
                       └──────────┬───────────┘
                                  │ WS · /api/chat/ws · AppEvent
                                  │
              ┌───────────────────┼───────────────────┐
              ▼                   │                   ▼
   ┌─────────────────┐    ┌──────┴────────┐   ┌──────────────────┐
   │    ironclaw     │    │ jarvis_voice_ │   │ ElevenLabs Conv. │
   │  (fork core)    │    │    daemon     │←──┤      (cloud)     │
   │  Rust headless  │    │ Rust · cpal   │   │      WSS         │
   │  systemd-user   │    │  systemd-user │   └──────────────────┘
   └────────┬────────┘    └───────┬───────┘
            │ in-process (Tool trait)
            ▼
   ┌─────────────────┐
   │ jarvis_system_  │   ← renombrado de jarvis_linux_mcp/{tools,adapters}
   │     tools       │     SIN server stdio · registra tools en startup
   │ workspace crate │
   └─────────────────┘
```

- **`jarvis_ui`** (NEW) — único proceso QML, layer-shell, 4 surfaces (1 ring + 3 widgets).
- **`ironclaw`** (FORK CORE) — sin cambios estructurales. Suma `jarvis_system_tools` al registry builtin.
- **`jarvis_voice_daemon`** (UNCHANGED) — sigue tal cual hoy. Integración con engine event loop (transcripts → AppEvent) es trabajo de v1.1, no v1.
- **`jarvis_system_tools`** (RENAMED) — re-empacado del contenido útil de `jarvis_linux_mcp`. Tools y adapters cambian su trait impl al `ironclaw::tools::tool::Tool`. Sin `mcp_server.rs` ni `bin/main.rs`.

`jarvis_policies` (sin cambios estructurales) se conecta como pre-execution hook del safety pipeline para las tools sistémicas, ya no llamado desde el MCP server (que desaparece).

## UI: layout, smart visibility, hotkeys

### Surfaces v1

| Surface | Posición fija | Visibilidad |
|---|---|---|
| `ring` (orbe) | bottom-right anchor | siempre visible — presencia de Jarvis |
| `tools_active` | top-right anchor | aparece con `tool_started`, fade out 3s tras último tool en estado terminal |
| `thread_cost` | top-left anchor | aparece con `thinking`/turn iniciado, fade out 5s tras último `response` |
| `events_stream` | bottom-left anchor | solo override hotkey (debug-friendly, no auto-aparece) |

**Postpuestos a v1.1**: `voice_transcript` widget, ring reactivo a TTS volume / wake-word state. Ambos requieren cablear voice_daemon ↔ engine event loop.

### Smart visibility

Default: orbe siempre visible (estado idle = glow suave + rotación lenta), widgets aparecen automáticamente cuando IronClaw emite eventos relevantes y desaparecen tras N segundos sin actividad. Posiciones fijas v1; drag-persisting v1.1.

### Hotkeys v1

| Hotkey | Efecto |
|---|---|
| `Super+J` (toggle) | Override: muestra los 3 widgets (incluido `events_stream`) aunque no haya actividad. Auto-hide tras 10s sin trigger nuevo. Re-toggle apaga inmediato. |
| `Super+Shift+J` | Force-hide: oculta todos los widgets aunque haya actividad. Útil para presentaciones / screenshots. El orbe sigue visible. |

### Estados visuales del orbe

- **Idle**: opacity 0.85, rotación lenta (14s), glow suave.
- **Active** (turn en progreso o tool ejecutando): rotación rápida (4s), glow más intenso, brillo ámbar puntual al activarse.
- **Offline** (gateway IronClaw no responde): tinte ámbar, sin animación, label `OFFLINE`.

## Comunicación con IronClaw

### Endpoints

| Tipo | URL | Uso |
|---|---|---|
| WebSocket | `ws://127.0.0.1:8080/api/chat/ws` | stream principal de AppEvent + control plane bidireccional |
| REST GET | `/api/threads`, `/api/jobs` | snapshot inicial al boot de jarvis_ui |

`HTTP_PORT=8080` por default según `.env.example`. Override por env var si el user cambia el puerto.

### Mapping AppEvent → widget

| AppEvent variant | Widgets que consumen |
|---|---|
| `tool_started` | `tools_active`, `ring` (intensifica), `events_stream` |
| `tool_completed` | `tools_active`, `ring` (relaja), `events_stream` |
| `tool_result` | `events_stream` |
| `thinking` | `thread_cost`, `ring`, `events_stream` |
| `response` | `thread_cost`, `events_stream` |
| `stream_chunk` | `thread_cost` (heuristic count para token estimado v1) |
| `status` | `events_stream` |
| `job_started` / `approval_needed` / `gate_*` | `events_stream` (visibilidad pasiva v1; widgets dedicados v2+) |

### Decisiones operativas

- **Transport**: 1 conexión WebSocket persistente desde QML `WebSocket` import nativo (Qt WebSockets). Cada widget consume signals del `EventBus.qml`, no abre su propia conexión.
- **Auth v1**: loopback only + Origin check (igual contrato que la web UI hoy). Sin token. v1.x suma bearer si se expone LAN.
- **Reconnection**: exponential backoff 1s → 30s cap. Estado offline visible en orbe (tinte ámbar). Reset de state al reconectar.
- **Snapshot inicial**: GET REST a `/api/threads` y `/api/jobs` para hidratar `thread_cost`. Tools en ejecución no tienen endpoint snapshot hoy → `tools_active` arranca vacío y se llena con stream (edge case aceptable: tool en ejecución pre-arranque de UI es raro).
- **Resilience**: si gateway está down, UI sigue corriendo, orbe muestra estado offline, widgets ocultos, reconnect en background.
- **Backpressure**: `events_stream` tail bounded a últimos 50. `tools_active` limpia tools terminadas tras 3s. `thread_cost` descarta turns más viejos que el thread activo.

### Caveat reconocido

`cost ($)` preciso no está en AppEvent. v1 muestra `turns + tokens estimado` (tokens calculados como heuristic sobre `stream_chunk` count). Para `$cost` real hay dos opciones, decisión queda en writing-plans:

- (a) GET REST a `/api/threads/:id` cada 5s para refrescar cost — depende de que el endpoint exponga el campo.
- (b) Agregar variant a `AppEvent::TurnComplete { cost_usd, ... }` — toca core upstream, divergencia explícita.

## Stack técnico, repo layout, build/deploy

### Repo layout

```
jarvis-os/
├── crates/
│   ├── ironclaw_*                    # core upstream + extensiones in-tree
│   ├── jarvis_voice_daemon/          # sin cambios
│   ├── jarvis_policies/              # sin cambios estructurales
│   └── jarvis_system_tools/          # NEW — renombrado de jarvis_linux_mcp
├── src/
│   └── tools/builtin/                # registra system_* tools al startup
├── ui/                               # NEW — Quickshell QML, NO crate Rust
│   └── jarvis-os/
│       ├── shell.qml                 # entry · registra surfaces layer-shell
│       ├── theme/{Palette,Typography,Animations}.qml
│       ├── core/{EventBus,IronclawRest,Hotkeys}.qml
```

`ui/` vive en raíz del repo (no en `crates/`) porque QML no es crate Rust. Sin `Cargo.toml`. Sin compilación.

### Dependencias del runtime

| Paquete | Origen | Propósito |
|---|---|---|

| `qt6-websockets` | Pacman | WebSocket cliente nativo desde QML |

### Comandos

| Acción | Comando |
|---|---|

## Lo que NO se toca

- `crates/ironclaw_*` — core upstream sin cambios. Pull de `staging` sigue trivial.
- `crates/jarvis_voice_daemon/` — funciona contra ElevenLabs hoy, sin cambios v1.
- El cambio en `crates/jarvis_voice_daemon/src/audio.rs` que está sin commitear — orthogonal, se maneja aparte.
- `arch/install.sh` base (Hyprland, end-4, paru, snapper) — solo se le agregan las líneas de UI install en M3 y M7.
- Las decisiones cerradas del 2026-04-29: voice daemon en Rust, ElevenLabs Convai cloud, PipeWire echo-cancel, target = Asus ZenBook UX431FLC.

## No-objetivos v1

- **Voice transcript widget** y **ring reactivo a TTS** → v1.1 cuando se cablée voice_daemon ↔ engine event loop.
- **Browser automation** (port de OpenClaw) → v1.x como tool builtin del fork. NO en v1; v1 es la UI cosmechándose con la actividad existente de IronClaw.
- **Computer-use desktop nativo** (xdotool/ydotool/Anthropic Computer Use) → v2+. Scope explícito de "navegar / hacer registro" se cubre con browser only.
- **Drag + persistencia de layout** → v1.1.
- **Soporte multi-monitor** → v1.x. v1 asume single-monitor primario.
- **Soporte GNOME/Mutter, KDE/KWin, X11** → no en roadmap. Wayland wlroots-only.
- **Marketplace de skins/temas** → no en roadmap.
- **`$cost` exacto en `thread_cost`** → v1 muestra turns + tokens estimado. Decisión de cómo obtenerlo (REST o AppEvent variant nuevo) queda para writing-plans.

## Decisiones cerradas (no re-debatir)

| Decisión | Razón |
|---|---|
| jarvis-os = UI sobre IronClaw, no OS | El chat ya está en IronClaw; lo distintivo es voz + visual |
| Stack UI = Quickshell + QML puro | Diseñado para shells wlroots; mejor render práctico que Tauri WebView; el target ya tiene Quickshell instalado vía end-4 |
| Sin segundo stack para el ring | Escape hatch a `QQuickItem` C++ dentro del mismo proceso si QML puro no rinde — empírico |
| MCP local fuera | Las tools sistémicas se cubren con `shell` o se reescriben como builtin; ya es fork |
| `ui/` en raíz, no en `crates/` | QML no es crate Rust |
| Wayland wlroots-only | wlr-layer-shell es la base de la UI |
| Browser ≠ computer-use desktop nativo | Scope reducido, riesgo controlado, OpenClaw cubre browser |
| Smart visibility + override hotkey | Ambient + glanceable + control manual cuando se necesita |
| Posición fija v1 | Drag complejo, no vale la pena pre-build |
| WebSocket sobre SSE | Bidireccional, futuro-proof, mejor para alta frecuencia |

## Cuestiones abiertas para writing-plans

- **M2 sub-spec**: la conversión del trait impl + safety hooks puede merecer su propio sub-spec. Decidir en writing-plans si es 1 milestone o 2-3.
- **M4 sub-spec**: integración de `jarvis_policies` al safety pipeline. Hay dos call sites posibles (dentro de `jarvis_system_tools` por tool, o como capa transversal de `ironclaw_safety`). Decidir cuál.
- **`$cost` para `thread_cost`**: REST poll cada 5s, o agregar `AppEvent::TurnComplete { cost_usd }`. Decidir en plan.
- **Estructura de PRs**: 1 PR único, 3 PRs (Track A, Track B, merge), o más granular (1 PR por milestone). Decidir en plan.
- **Tests v1**: nivel mínimo viable (smoke + qmllint) o sumar `qmltest` para componentes con state desde el día 1. Decidir en plan según costo.
- **`shell.qml` boilerplate**: layer-shell anchors exactos, exclusion zones, namespace de cada surface. Detalles concretos para writing-plans.
- **Hotkey binding en Quickshell**: si el `GlobalShortcut` Qt no funciona en wlroots, alternativa via `niri-msg`/`hyprctl dispatch` con keybind del compositor. Validar en M6.b.

Estos no bloquean el plan; se resuelven en el momento de escribir el código de cada paso M1-M8.
